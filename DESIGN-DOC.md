## Problem Statement

The motorcycle rental business is established and healthy: 30–50 rentals per month,
concentrated almost entirely into weekends, with up to 10+ rentals in a single
Saturday–Sunday. All of it is currently coordinated through messaging plus the
manager's memory and a paper calendar.

There is no single source of truth for which bike is free when. Two recurring
failures result:

1. **Double-bookings.** 10+ overlapping weekend reservations cannot be reliably
   tracked in memory and on paper. When two bookings collide, the cost is a refund,
   an angry renter, and reputation damage.
2. **Availability-question overload.** A constant stream of "is this bike free on
   Saturday?" messages, each answered manually, one at a time, burning the manager's
   evenings.

The website exists to give renters a way to see, themselves, which specific
motorcycle is free at what time — and to give the manager one reliable place to
record bookings.

## Demand Evidence

This is an established operation, not a hypothesis. In the founder's own words:

> "only rent on weekends, but can up to 10+ every weekend, so about 30-50 per month"

That is real, recurring, paying demand for motorcycle rentals.

Important distinction: demand for the rental **service** is proven. Demand for the
**website** is not yet proven — it is the thing version one exists to test. The
signal that the website is genuinely needed: renters referencing the page instead
of texting ("I saw the Ducati is free Saturday"), and the manager going a full
month with zero double-bookings.

## Status Quo

Booking coordination today:

1. A renter messages or texts the manager.
2. The manager checks a paper calendar or recalls what is already booked.
3. The manager replies.

Availability lives in the manager's head and on paper. At low volume this is fine.
At 10+ bookings crammed into one weekend it is fragile: memory and paper do not fail
gradually — they fail all at once, on the busiest weekend, when reservations overlap.

The competitor here is not another rental shop. It is this manual workaround. The
website only earns its keep if it beats the workaround on the two failures above.

## Target User & Narrowest Wedge

**Two users:**

- **The renter** — a weekend leisure rider who comes for a *specific* motorcycle (a
  particular model, the photogenic one) and is disappointed if it is taken. They
  want to know, fast and on a phone, whether that bike is free for the weekend they
  have in mind.
- **The manager** — the founder, who needs one trustworthy place to see and record
  what is booked.

**The narrowest wedge** (chosen deliberately over a fuller self-serve booking
platform): a **read-only, manager-maintained availability page.**

- The manager enters bookings. Renters only view.
- Per-bike, with photos — because renters choose a specific bike.
- Renters still text or call to actually book — but only *after* seeing the bike is
  free, which removes the "is it available?" rounds entirely.

**Explicitly out of scope for version one:** self-serve booking, payments, deposits,
damage waivers, renter accounts/logins, detailed spec sheets, reviews, search,
sub-day time slots. See the NOT in Scope section.

## Constraints

- **Speed:** must be live before the next busy weekend.
- **Solo operator:** the manager is the only person running this. The tool cannot
  assume engineering staff or significant ongoing maintenance effort.
- **Scale:** trivial. With build-time data baking (see Architecture), renters load
  only static CDN files — 50–100 concurrent views is a non-issue and there is zero
  API rate-limit exposure at request time.
- **Mobile-first:** renters book last-minute, on phones.
- **Low/zero ongoing cost** preferred — this is a small operation.

## Premises

All six were confirmed by the founder.

1. **Root cause** is no single source of truth for availability, not "no website."
   Fix the source of truth and both symptoms (double-bookings, texts) die together.
2. **Build vs buy.** A small custom build is justified for the read-only wedge —
   free tools render per-bike-with-photos availability badly (a shared Google
   Calendar cannot show six motorcycles as six photo cards with weekend free/busy).
   BUT self-serve booking, payments, and deposits should be re-evaluated against
   off-the-shelf rental software (Booqable, Rentle) rather than built. The danger:
   "I built the page" quietly becoming "I am building a rental platform."
3. **Scope.** The wedge is correctly scoped. Self-serve, payments, accounts, specs
   are later, not now. Version one ships in days.
4. **Cost of nothing** is real and recurring (double-bookings, lost evenings) —
   current, not hypothetical.
5. **Distribution** is trivial — one URL shared in existing channels.
6. **Fleet shape:** roughly 4–10 motorcycles; bookings are full-day or multi-day
   weekend rentals, not by-the-minute.

## Approaches Considered

### Approach A — Spreadsheet brain, static face *(minimal viable — CHOSEN)*
Airtable holds bikes and bookings; the manager edits it from the Airtable phone app.
A small build step bakes that data into a static page, refreshed by a ~15-minute
scheduled rebuild, which renders per-bike availability with photos.
Effort: S (human ~2 days). Risk: Low.

### Approach B — A real small app *(ideal architecture)*
A small web app with a real database. Public availability page + password-protected
admin page. Overlapping bookings are rejected by the server, not just flagged.
Effort: M (human ~3–6 days). Risk: Low–Med.
Rejected because: premise 2 caps this app at the read-only wedge permanently, so B's
architectural headroom serves a future that should go to Booqable instead. Its one
live advantage (prevent vs flag double-bookings) does not justify 2–3x the build.

### Approach C — The calendar IS the backend *(creative / lateral)*
No admin UI. The manager keeps using a calendar app; the website renders those
calendars as a polished availability view. Effort: S–M. Risk: Med.
Rejected because: calendar-feed plumbing (auth, refresh, time zones) is fiddly, and
bike photos/prices still need a separate store — so it does not actually remove much
build while adding production risk.

## Recommended Approach

**Approach A — Spreadsheet brain, static face.**

Rationale: the website must be live before the next busy weekend, and premise 2
means this app stays the read-only wedge permanently. The real leap is going from
memory-and-paper to one shared source of truth that flags overlaps. Approach B's
prevent-vs-flag gain is marginal next to that leap and costs 2–3x the time. If
double-bookings still slip through after a month on Approach A, that is the signal
to move the whole operation to Booqable — not to rebuild as B.

### Architecture

Data is fetched from Airtable **at build time, never in the browser.** The renter's
page is 100% static files, rebuilt on a schedule.

```
 Manager          Build pipeline (scheduled, ~every 15 min)        Renter
 ───────          ─────────────────────────────────────           ──────
 edits Airtable   1. fetch Bikes + Bookings (token                 opens a URL
 from the phone      held only as a build secret)                       │
 app               2. download + resize bike photos                    ▼
        │          3. validate bookings                             static page
        ▼          4. compute weekend window                        from the CDN
   Airtable base   5. compute per-day free/busy                     (0 Airtable
   (Bikes,         6. compute overlap flags                          calls, 0
    Bookings)      7. write public-fields-only data                  secrets,
        │          8. generate index.html → deploy                   loads <2s)
        │                          │
        └─ (no live link) ─────────┘
                          GitHub Actions → GitHub Pages
```

- **Backend:** none. Airtable is the entire data store.
- **Manager UI:** none built. The manager uses the Airtable mobile app — a real
  structured editor with a calendar view, for free.
- **Renter UI:** one static page (HTML/CSS/JS). No login, no accounts.
- **Build tool — plain Node script, no framework** (decided in eng review). The
  project is one page; a static-site generator's data-cascade and plugin machinery
  would be accidental complexity. File layout:
  - `build.mjs` — orchestrator. Does **all** I/O: fetch Airtable, download photos,
    write files. The only file that touches the network or disk.
  - `lib/dates.mjs`, `lib/availability.mjs`, `lib/project.mjs`, `lib/render.mjs` —
    pure functions, no I/O, unit-testable in isolation.
  - `index.template.html` + `styles.css` — page template and styling.
  - `assets/placeholder.svg` — missing-photo fallback.
  - `.github/workflows/build.yml` — the scheduled build.
- **Build trigger — one scheduled GitHub Actions workflow, ~every 15 minutes**
  (decided in eng review). One trigger, one secret, no Airtable automations. An
  instant on-change trigger was considered and deferred — see NOT in Scope.
- **The Airtable token lives only in the build environment** as a CI secret. Never
  shipped to the browser. The build writes a data file containing only public
  fields; the private `Renter note` is read by the build but never written to any
  client-visible file.
- **Hosting:** GitHub Pages, on a **public** repo (free, unlimited Actions minutes).

### Why build-time, not browser-time

A browser-side fetch would put the Airtable token in client JavaScript, where any
visitor can extract it and read the entire base directly — including private renter
notes — and a per-browser cache does nothing to coalesce 50–100 distinct visitors
against Airtable's ~5 requests/second/base limit. Build-time baking eliminates all
of it: the token stays server-side, renters hit only the CDN, and the build makes
~2–4 Airtable calls per run, far under any limit.

The cost is staleness: the page reflects an Airtable edit only after the next
scheduled build, so it can be up to ~15 minutes behind. Know the direction of the
risk in that window:

- A **false FREE** (page still shows a just-booked bike as free) can produce one
  extra "is it available?" text, or a disappointed renter.
- A **false BOOKED** (page still shows a just-cancelled bike as booked) can lose a
  booking.

This is bounded and acceptable for v1: renters text to confirm every booking, so a
stale page degrades to the old texting pain for one query, never to a
double-booking. If 15-minute staleness ever genuinely bites, the deferred instant
trigger (see NOT in Scope) closes the window to ~2 minutes.

### Airtable schema

**Bikes table**
- `Name` (text) — e.g. "Ducati Monster"
- `Type` (single select) — Cruiser, Naked, Sport, Adventure, …
- `Engine` (text) — e.g. "937cc"
- `Price per day` (currency) — optional; the page shows it only if populated
- `Photo` (attachment) — should always be set (renters choose by specific bike);
  the build tolerates a missing one, see build logic step 2
- `Display order` (number) — controls ordering on the page
- `Active` (checkbox) — see "States and edge cases" for exact meaning

**Bookings table**
- `Bike` (link to Bikes)
- `Start` (date) — first day the bike is out, inclusive
- `End` (date) — last day the bike is out, inclusive
- `Renter note` (text) — manager's private note; read by the build but **never**
  written to any client-visible file

v1 is **day-level only.** `Start` and `End` are plain dates. Sub-day time slots are
out of scope. No `Status` field — a booking either exists or it does not.

### Build logic (runs in CI on a ~15-minute schedule)

1. Fetch Bikes; keep `Active = true`; sort by `Display order`. Fetch Bookings
   (optionally only those with `End >= today` to keep payload small).
2. For each kept bike, download its `Photo`, resize to ~800px wide, save it into
   the static output. The page references this local copy — **not** the Airtable
   attachment URL, which expires. If a bike has no photo or the download fails:
   substitute a built-in placeholder and log a warning. One bad row must never fail
   the whole build.
3. **Validate each booking.** A booking is malformed if `Start` or `End` is missing
   or unparseable, or `End` is before `Start`. Drop each malformed booking from the
   availability calculation, attach a reason-carrying warning to its bike, and log
   a build warning. The build continues — one bad row never fails it.
4. Build the display window: the next 4 weekends (4 Saturday–Sunday pairs) from the
   build date, in the business's local timezone (a hard-coded config constant).
5. For each bike × each weekend day: the day is **BOOKED** if any *valid* booking
   for that bike satisfies the interval test `Start <= day AND End >= day`
   (inclusive — handles long bookings straddling the window edge). Otherwise FREE.
6. For each bike, set an overlap flag if any two of its valid bookings intersect
   (`A.Start <= B.End AND B.Start <= A.End`). This drives the `⚠` marker.
7. Write a static data file containing only public fields (no `Renter note`).
8. Generate `index.html` from the template + this data, stamped with the build time.
9. Deploy.

### States and edge cases

- **No active bikes:** page shows "Fleet list coming soon."
- **A bike with no bookings in the window:** all four days show FREE. Normal.
- **No bookings at all:** every bike shows all-FREE. Normal.
- **Missing/broken bike photo:** placeholder image, build continues (step 2).
- **Malformed booking** (bad/missing date, `End` before `Start`): the booking is
  dropped from the calculation and its bike shows a reason-carrying `⚠` ("a booking
  has a bad date"). Build continues (step 3).
- **Build fails** (e.g. Airtable unreachable at build time): the last successful
  deploy stays live. The page keeps working; its "updated HH:MM" stamp ages.
- **First build:** there is no "last successful deploy" to fall back to. The first
  build must be confirmed successful and visually checked **before the URL is
  shared** with any renter.
- **Stale page:** the build timestamp is always visible. The manager treats a
  timestamp older than **~45 minutes** as a broken-pipeline alarm (45 minutes
  exceeds three missed 15-minute builds plus CI jitter, so it does not cry wolf).
- **A broken pipeline also freezes overlap detection.** The `⚠` flags are
  recomputed only on a successful build. While the pipeline is down, a
  double-booking or malformed row entered in Airtable will not be flagged. The
  staleness alarm and the `⚠` are therefore the same safety signal — heed it.
- **`Active` checkbox meaning:** `Active` retires a bike from the fleet entirely.
  Do **not** use it to mark a bike unavailable for a weekend. To block a bike
  (service, personal use), create a normal booking for it. Do not deactivate a bike
  that still has upcoming bookings shown on the page.
- **Long / multi-week bookings:** handled correctly by the inclusive interval test
  in build logic step 5; a booking partially overlapping the window marks only its
  in-window days.

### Double-booking handling — be honest about the limit

Approach A **flags**, it does not **prevent**.

- **At entry time:** the manager uses Airtable's per-bike Calendar view, where two
  overlapping bookings render as overlapping blocks — visually catchable.
- **On the page:** overlapping bookings for one bike surface a `⚠` warning, within
  ~15 minutes of entry (the next scheduled build). If the build pipeline is down,
  no new `⚠` appears until it recovers — so the staleness alarm must be treated as
  a real alarm.
- It **cannot** stop the manager typing a double-booking into Airtable. That hard
  guarantee is what Approach B (or Booqable) would add. Accepting this limit is a
  deliberate trade for shipping in days instead of weeks.

### Privacy note

The public page and the baked data file contain only FREE / BOOKED state. Renter
names, notes, and contact details never reach the browser — the `Renter note` field
is excluded by the build script (build logic step 7), and there is no API token in
the client to bypass that. The read token itself is base-wide and *can* read the
field (see Dependencies); the guarantee rests entirely on that build-step exclusion,
which holds *because* of the build-time architecture. A unit test and an integration
test both assert it (see Testing).

### Testing

Test runner: Node's built-in **`node:test`** (requires Node 20+). Zero test
dependencies. Run with `node --test`. Decided in eng review: full unit coverage of
the `lib/` pure functions plus one integration test of the whole build. ~28 paths;
all must be covered before ship. No regression tests — greenfield, nothing to break.

Required tests:

- **`lib/dates.mjs` — `weekendWindow()`:** weekday build date, build date on
  Sat/Sun, window crossing month and year boundaries, a DST transition inside the
  window.
- **`lib/availability.mjs`:**
  - `dayStatus()` — no bookings (FREE), booking covers the day (BOOKED), single-day
    booking, booking straddling the window edge, multi-week booking.
  - `hasOverlap()` — zero/one booking, two bookings touching on one inclusive day
    (IS an overlap), genuine overlap, three bookings where only two overlap.
  - `validateBooking()` — missing/unparseable `Start` or `End`, `End` before
    `Start`, a valid row.
- **`lib/project.mjs` — `toPublicData()`:**
  - **★ CRITICAL (security):** assert the output contains NO `Renter note` and no
    other private field. The entire privacy guarantee rests on this test.
  - malformed booking excluded and its bike flagged (build logic step 3).
  - output shape: per-bike, per-day FREE/BOOKED plus warning flags.
- **`lib/render.mjs` — `renderPage()`:** zero active bikes ("Fleet list coming
  soon"), zero bookings (all FREE), a bike with an overlap `⚠`, a bike with a
  bad-data `⚠`, a bike with no photo (placeholder), price present vs absent, the
  build timestamp rendered.
- **`build.mjs` — integration:** a full build against a saved fixture Airtable
  response produces correct HTML and data file; a fixture with a missing photo
  yields the placeholder and the build still succeeds. The integration test also
  re-asserts end-to-end that no `Renter note` appears anywhere in the output.

Not unit-tested: the GitHub Actions workflow and the live deploy — covered by the
manual first-build verification, and later by `/qa` for mobile layout.

## Success Criteria

- Version one is **live at a public URL before the next busy weekend.**
- The manager records **100% of bookings** in Airtable for one full month — the
  single source of truth is actually single.
- **Zero double-bookings** across one month of use (measurable, binary).
- A **noticeable drop in "is bike X free?" messages** — target is most renters
  referencing the page when they make contact.
- The availability page **loads in under 2 seconds on a phone** (static CDN files
  plus build-time-resized images, no runtime API calls).
- The page shows a **last-updated timestamp**; a stamp older than ~45 minutes is
  the manager's signal that the build pipeline is broken (and that `⚠` detection is
  frozen until it is fixed).
- **All ~28 test paths in the Testing section pass** before the URL is shared.
- **Trigger condition:** if double-bookings still occur after one month, that fires
  the build-vs-buy re-evaluation — move to Booqable, or to Approach B.

## Distribution Plan

This is a static web page. Distribution = hosting + a link.

- **Host:** GitHub Pages, free tier.
- **Build/deploy pipeline:** the build script runs in CI on a **single GitHub
  Actions scheduled workflow, ~every 15 minutes.** Each run fetches Airtable, bakes
  data and images into static files, and deploys to GitHub Pages. The Airtable
  token is stored as a GitHub Actions secret — the only place it exists.
  - Use a **public repo:** GitHub Actions is unlimited-free for public repos, and
    the Airtable token is a CI secret never in the code, so public is safe.
  - **Deferred:** an instant on-change trigger (Airtable automation → GitHub
    `repository_dispatch`) would cut staleness to ~2 minutes, but it needs a second
    credential (a GitHub token stored inside Airtable) and carries Airtable's
    ~100/month free automation cap. Not worth it for v1 — see NOT in Scope.
- **Domain:** optional custom domain (~$10–15/year). v1 uses the free
  `*.github.io` URL.
- **Sharing:** paste the URL into the channels renters already use — text replies,
  WhatsApp, Instagram bio and Stories. Share it only after the first build is
  confirmed live.
- No app store, no installs.

## Cost

The whole design runs on free tiers. Realistic monthly cost for v1: **$0.**

| Piece | Free tier covers it | Caveat |
|---|---|---|
| Airtable — data store | 2 tables, ~1,000 records/base, API reads, 1 GB attachments | The 1,000-record cap is ~1 year of runway at your volume; delete past bookings periodically |
| GitHub Actions — build | Unlimited for a **public** repo | A private repo gets only 2,000 min/month free; use a public repo (the Airtable token is a CI secret, never in the code, so public is safe) |
| GitHub Pages — hosting | Free | — |
| Custom domain | — | Optional, ~$10–15/year; skip it and use the free `*.github.io` URL |
| Airtable automations | Not used in v1 | Only the deferred instant trigger would need them (~100 runs/month cap). v1's ~15-minute scheduled build uses none |

Because v1 uses only the scheduled build, there is no automation cap to hit and no
paid path in play. The only way this design would ever cost money is the deferred
instant trigger at high edit volume (Airtable Team plan, ~$20/month) — not part of
v1, and very unlikely to be worth it.

## Dependencies

- **Airtable account** (free tier) with one base: Bikes table + Bookings table per
  the schema above.
  - Free-tier record cap is ~1,000 records per base. At 30–50 bookings/month that
    is over a year of runway; periodically delete past bookings.
- **Bike photos** — the manager must supply these. A missing one degrades to a
  placeholder rather than breaking the build.
- **Airtable API token** — read scope. Airtable read tokens are **base-wide**; the
  token *can* read every field, including `Renter note`. That field's privacy is
  guaranteed solely by the build script excluding it (build logic step 7), not by
  token scoping. Stored as a **GitHub Actions secret** — never in client code.
- **A public GitHub repo** with GitHub Actions (unlimited-free) and GitHub Pages.
- **Node 20+** for the build script and `node:test`.
- Optional: a custom domain.

- **Photo-processing cache** — the build re-downloads/resizes all photos each run;

  the caching logic.
